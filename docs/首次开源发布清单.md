# 首次开源发布清单

这份清单面向当前仓库的首次公开版本，目标是把项目作为一个可参考、可复现、可继续迭代的初始版本发布出去。

## 一、这次发布的建议定位

建议把当前版本定义为：

- 版本号：`v0.1.0`
- 版本性质：初始稳定公开版
- 发布目标：帮助遇到类似坏键问题的人快速理解思路，并在自己的机器上做最小改造

建议你在仓库介绍里明确说明：

1. 这个工具最初是为 `ROG Zephyrus G14 2021` 的坏键问题而写
2. 当前首版先聚焦稳定和可用
3. 后续版本会增加自定义映射和双排布局

## 二、当前仓库已经具备的公开条件

当前仓库已经具备这些基础条件：

- 有完整源码
- 有首页说明文档
- 有测试与开发记录文档
- 已配置 `git（版本控制）`
- 已打首个版本标签
- 已忽略构建产物目录

## 三、还需要注意的发布项

发布前重点确认：

1. 首页说明是否足够让陌生用户看懂项目用途
2. 是否附带合适的开源许可证
3. 是否需要上传首版可执行压缩包
4. 是否要在仓库描述里明确“适用机型”和“当前限制”

## 四、推荐的仓库介绍文案

### 中文简述

一个面向华硕 `ROG` 笔记本坏键场景的轻量级 `Windows（微软桌面操作系统）` 工具，提供按键替代、屏幕亮度控制和键盘背光控制能力。

### 中文长描述

RogKeyMini 是我为 `ROG Zephyrus G14 2021` 键盘部分按键失效场景编写的一个小工具。它的目标不是做成大而全的键盘管理软件，而是在“少数实体按键损坏，但整机仍希望继续使用”的情况下，提供一个轻量、直接、可继续扩展的解决方案。当前公开的首个版本优先保证稳定与可参考性，后续会继续补充自定义坏键映射和更合适的双排界面布局。

## 五、推荐的开源许可

当前最适合这个仓库的是 `MIT（宽松开源许可）`：

- 允许别人学习、复用和修改
- 适合“给别人参考”的发布目标
- 不会像强约束许可证那样增加理解门槛

如果你未来明确希望别人修改后也必须继续开源，再考虑切换到更强约束的许可证。

## 六、当前远端信息

当前已经明确的远端地址：

- `GitHub（代码托管平台）`：<https://github.com/jieying27889-dotcom/ROG-Key-Mini>

如果后续你补充 `Gitee（代码托管平台）` 地址，建议远端命名：

- `gitee` 指向 `Gitee（代码托管平台）`

## 七、推送命令

### 推送到 `GitHub（代码托管平台）`

```powershell
git remote add origin https://github.com/jieying27889-dotcom/ROG-Key-Mini.git
git push -u origin main
git push origin v0.1.0
```

### 后续推送到 `Gitee（代码托管平台）`

```powershell
git remote add gitee https://gitee.com/<你的用户名>/ROG-Key-Mini.git
git push -u gitee main
git push gitee v0.1.0
```

## 八、是否建议上传可执行文件

建议上传一个首版压缩包到 `Release（发行包）` 页面，至少包含：

- `RogKeyMini.exe`
- 运行所需依赖文件
- `config.json`
- 简短使用说明

推荐压缩包名称：

`RogKeyMini-v0.1.0-win-x64.zip`

这样即使别人不看源码，也能先快速试用。

## 九、这次发布时如何描述后续计划

不要把未来计划混进当前版本能力说明里，建议单独列成路线图：

1. `v0.1.0`：固定坏键场景下的稳定替代操作
2. `v0.2.0`：支持自定义坏键与快捷键映射
3. `v0.3.0`：优化界面布局，调整为双排按钮布局

## 十、最适合你现在的执行顺序

建议按下面顺序推进：

1. 保持当前代码范围不再扩张
2. 先把文档、许可证和仓库介绍整理好
3. 推送源码和首个标签
4. 如果方便，再补一个首版压缩包
5. 后续新功能单独作为新版本继续迭代

对这个项目来说，最重要的不是一次把所有想法都做完，而是先把已经稳定的部分公开出去。
