# 首次开源发布清单

这份清单面向当前仓库的首次公开版本，目标是把项目作为一个可参考、可复现、可继续迭代的初始版本发布到 `GitHub` 和 `Gitee`。

## 一、先确定这次发布的定位

建议把这次版本定义为：

- 版本号：`v0.1.0`
- 定位：初始稳定版本
- 目标：解决特定机型上少数物理按键失效后的替代输入问题
- 说明方式：当前版本先聚焦“可用”和“稳定”，后续版本再做“可配置”和“界面优化”

建议你在仓库介绍里明确写清楚两件事：

1. 这个项目当前主要服务于你这台 `ROG Zephyrus G14 2021`
2. 后续计划会增加“自定义失效键映射”和“两排按钮布局”

## 二、发布前检查

当前仓库已经有这些优点：

- 已有 `README.md`
- 已有源码目录和文档目录
- 已有 `.gitignore`
- 构建产物 `artifacts/` 已被忽略，不会误传

发布前你只需要再确认下面几项：

1. `README.md` 的内容能让陌生人一眼看懂项目用途
2. 不把本地日志、临时文件、测试产物提交进去
3. 决定是否添加 `LICENSE`
4. 决定是否上传首版可执行压缩包作为 `Release`

## 三、推荐的仓库命名

建议二选一：

1. `rog-key-mini`
2. `rog-key-mini-g14`

如果你更希望别人一眼看出用途，也可以用：

1. `rog-broken-key-helper`
2. `asus-keyboard-workaround`

当前项目名 `ROG Key Mini2` 更像本地工作目录名，不太适合作为公开仓库名。

## 四、推荐的仓库简介

你可以直接用下面这段：

`A small Windows utility for ASUS ROG laptops with broken keyboard keys, focused on hotkey replacement and brightness/backlight control.`

如果放中文简介，可以写：

`一个面向华硕 ROG 笔记本坏键场景的轻量级 Windows 工具，提供按键替代、屏幕亮度和键盘背光控制。`

## 五、建议的发布顺序

先做本地 `git` 初始化，再同时推送到两个远端。

建议远端命名：

- `origin` 指向 `GitHub`
- `gitee` 指向 `Gitee`

这样后续维护最清楚。

## 六、本地初始化命令

在项目根目录执行：

```powershell
git init -b main
git add .
git commit -m "chore: prepare initial public release v0.1.0"
```

如果你想先看会提交什么，再执行：

```powershell
git status
```

## 七、创建远端仓库

### `GitHub`

1. 新建一个空仓库
2. 不要勾选自动创建 `README`、`.gitignore`、`LICENSE`
3. 仓库名使用前面建议的公开名

### `Gitee`

1. 同样新建一个空仓库
2. 也不要初始化任何默认文件
3. 仓库名尽量与 `GitHub` 保持一致

## 八、绑定远端并推送

把下面的地址替换成你自己的用户名和仓库名：

```powershell
git remote add origin https://github.com/<你的用户名>/<仓库名>.git
git remote add gitee https://gitee.com/<你的用户名>/<仓库名>.git
git push -u origin main
git push -u gitee main
```

如果你使用的是 `SSH`，也可以改成 `SSH` 地址。

## 九、给首版打标签

建议在首个公开版本就打标签，这样以后增加新功能时，别人还能明确找到你现在这个稳定起点。

```powershell
git tag -a v0.1.0 -m "Initial stable public release"
git push origin v0.1.0
git push gitee v0.1.0
```

## 十、是否要上传可执行文件

建议上传一个便携版压缩包到 `GitHub Release` 或 `Gitee Release`，文件里至少包含：

- `RogKeyMini.exe`
- 运行所需依赖
- `config.json`
- 简短使用说明

建议压缩包命名：

`RogKeyMini-v0.1.0-win-x64.zip`

这样不懂源码的人也能直接试用。

## 十一、这次发布时应该怎么描述未来功能

不要把未来计划混进当前版本说明里，而是单独放到“路线图”里。

推荐写法：

1. `v0.1.0`：提供固定坏键场景下的稳定替代操作
2. `v0.2.0`：支持自定义坏键与快捷键映射
3. `v0.3.0`：优化界面布局，改为更紧凑的双排按钮布局

这样做的好处是：

- 首版范围清晰
- 用户知道后面会继续做
- 你后续重构前端时不会影响首版定位

## 十二、建议的发布文案

可以直接作为提交说明、版本说明或仓库首页补充文本：

> RogKeyMini is my personal workaround project for a 2021 ROG Zephyrus G14 with several failed keyboard keys. I am publishing the initial stable version first so that other users with similar hardware problems can use it as a reference. Future updates will focus on custom key remapping and a better two-row interface layout.

## 十三、你现在最适合的执行方案

按优先级建议如下：

1. 先发布 `v0.1.0`
2. 保持当前功能范围不再扩张
3. 上传源码
4. 如果方便，再补一个首版压缩包
5. 后续新功能单独开新版本推进

对你这个项目来说，最重要的不是一次做全，而是先把“已经验证过、已经稳定”的部分公开出去。
