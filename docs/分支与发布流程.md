# 分支与发布流程

这份文档用于固定当前仓库后续的开发方式，目标很简单：

- 继续开发升级版本时，不影响已经公开的稳定版本
- 保留一个可以放心继续试验和重构的开发分支
- 让每次稳定发布都有明确的版本落点

## 当前分支约定

仓库使用两个长期分支：

### `main（主分支）`

用途：

- 保存稳定版本
- 对外公开展示
- 作为每次正式发布的基础

要求：

- 不直接在这里做高风险实验
- 只有在功能验证通过后，才合并进入
- 每个稳定节点都可以打 `tag（版本标签）`

### `dev（开发分支）`

用途：

- 开发新功能
- 调整界面布局
- 进行试验性修改
- 做尚未确认稳定的重构

要求：

- 日常开发默认在这里进行
- 如果写坏了，只影响开发线，不影响稳定版
- 阶段完成后再决定是否合并到 `main（主分支）`

## 推荐开发流程

### 日常开发

1. 切到 `dev（开发分支）`
2. 在 `dev（开发分支）` 上开发和提交
3. 需要时从 `dev（开发分支）` 再切功能分支
4. 功能验证通过后，再合并回 `main（主分支）`

常用命令：

```powershell
git checkout dev
git pull
git checkout -b feature/custom-key-remap
```

### 发布稳定版

1. 确认 `dev（开发分支）` 上的功能已经稳定
2. 合并到 `main（主分支）`
3. 在 `main（主分支）` 上打新的 `tag（版本标签）`
4. 推送 `main（主分支）` 和新标签

常用命令：

```powershell
git checkout main
git merge dev
git tag -a v0.2.0 -m "Stable release v0.2.0"
git push origin main
git push origin v0.2.0
```

## 为什么这样做

这样做的核心价值是：

1. 已发布版本有固定落点，不会被后续开发污染。
2. 后续新功能可以放心试，不用担心把稳定版本一起带坏。
3. 如果某次开发方向不满意，可以只处理 `dev（开发分支）`，不用碰历史稳定标签。

## 对这个项目的具体建议

对 RogKeyMini 这个项目，后续更适合放到 `dev（开发分支）` 的内容包括：

1. 自定义坏键映射
2. 双排按钮布局
3. 更灵活的热键配置
4. 更多机型适配尝试

而更适合保留在 `main（主分支）` 的，是已经验证过的稳定替代方案。

## 额外建议

如果你后面主要在 `GitHub（代码托管平台）` 上维护，建议再手动做两件事：

1. 给 `main（主分支）` 开启保护，避免误提交和误强推。
2. 新功能优先通过 `Pull Request（合并请求）` 从 `dev（开发分支）` 合并到 `main（主分支）`。

这样稳定线会更安全。
